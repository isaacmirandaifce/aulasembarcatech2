## Processamento

# Arquitetura Básica de Firmware

Grande parte dos firmwares segue a mesma estrutura lógica.

1. Inicialização do hardware
2. Loop infinito
3. Leitura de sensores
4. Processamento
5. Controle de atuadores

Exemplo conceitual:

```c
int main() {

    inicializarHardware();

    while(1){
        lerSensores();
        processarDados();
        controlarSaidas();
    }

}
```

Esse padrão aparece em **quase todo sistema embarcado**.

# Problema Prático: Monitoramento de Temperatura

Imagine um sistema embarcado que monitora a temperatura de um equipamento.

Regras de operação:

* Temperatura < 40°C → **Sistema Normal**
* Temperatura entre 40°C e 60°C → **Alerta**
* Temperatura > 60°C → **Estado Crítico**

Pergunta:

**Como implementar essa lógica usando C?**

---

# Estruturas de Decisão

Para implementar decisões usamos:

* operadores relacionais
* operadores lógicos
* estruturas `if` e `else`

Exemplo:

```c
if(temperatura > 60){
   printf("CRITICO");
}
```

Esse tipo de lógica aparece em firmware para:

* proteção térmica
* monitoramento de tensão
* controle de motores
* proteção de baterias

---

# Desafio Prático 1

Escrever um programa que:

1. receba um valor de temperatura
2. avalie a condição do sistema
3. imprima o estado correspondente

Estruturas sugeridas:

* `if`
* `else if`
* `else`

# Possível Solução

```c
if(temp < 40)
    estado = 0;
else if(temp < 60)
    estado = 1;
else
    estado = 2;
```

Pergunta para reflexão:

Esse código funciona, mas pode ficar **confuso em sistemas maiores**.

Como organizar melhor a lógica?

# Máquinas de Estado

Sistemas embarcados frequentemente utilizam **máquinas de estado**.

Cada estado representa uma condição do sistema.

Exemplo:

* NORMAL
* ALERTA
* CRÍTICO

Transições entre estados dependem de sensores ou eventos.

---

# Uso do Switch

Estrutura muito usada em firmware.

```c
switch(estado){

case NORMAL:
   printf("Sistema normal");
break;

case ALERTA:
   printf("Sistema em alerta");
break;

case CRITICO:
   printf("Sistema crítico");
break;

}
```

Isso torna o código:

* mais organizado
* mais legível
* mais fácil de manter

---

# Loops no Contexto de Firmware

Estruturas de repetição possuem papéis diferentes.

## for

Usado quando sabemos o número de repetições.

Exemplo:

* coletar N leituras de sensor

## while

Execução contínua.

Muito usado em firmware.

## do while

Garante que o código execute pelo menos uma vez.

# Problema Real: Ruído em Sensores

Sensores físicos não produzem leituras perfeitas.

Problemas comuns:

* ruído elétrico
* interferência
* pequenas variações de medição

Uma técnica simples:

**tirar várias amostras e calcular média**

---

# Desafio Prático 2

Simular leitura de sensor.

Objetivo:

* ler **10 valores**
* calcular **média**
* imprimir o resultado

Ferramentas obrigatórias:

* loop `for`
* variável acumuladora
* divisão

---

# Exemplo de Implementação

```c
float soma = 0;

for(int i = 0; i < 10; i++){
   scanf("%f", &valor);
   soma += valor;
}

media = soma / 10;

printf("Media: %f", media);
```

Esse tipo de técnica aparece em:

* sensores analógicos
* filtragem de ruído
* processamento de sinais simples

---

# Problema de Escalabilidade

Firmware cresce rapidamente.

Problemas comuns:

* funções muito grandes
* código repetido
* difícil manutenção
* difícil depuração

Solução:

**dividir o sistema em funções**

---

# Arquitetura Modular

Separação em funções:

```c
float lerSensor();
float filtrarLeitura(float valor);
void controlarSistema(float temperatura);
```

Benefícios:

* modularidade
* reutilização
* organização
* manutenção mais simples

---

# Estrutura de Firmware

Exemplo típico:

```c
while(1){

   temperatura = lerSensor();

   temperatura = filtrarLeitura(temperatura);

   controlarSistema(temperatura);

}
```

Esse padrão aparece em:

* IoT
* automação
* robótica
* dispositivos médicos
